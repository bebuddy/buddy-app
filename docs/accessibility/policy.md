# 접근성 정책 (KS X 3253:2016 기반)

## 공통 원칙
- 모든 핵심 기능은 **단순 탭/클릭**으로 수행 가능해야 한다.
- 스크린리더(VoiceOver/TalkBack)에서 **이름/역할/상태**를 제공한다.
- 색만으로 상태를 전달하지 않는다(텍스트/아이콘/스크린리더 문구 병행).
- 중요한 알림은 **시각 + 스크린리더**로 반드시 인지 가능해야 한다.

## 인식의 용이성(Perceivable)
- 텍스트가 아닌 콘텐츠에 **대체 텍스트** 제공
  - 장식용 이미지는 `alt=""` 또는 `aria-hidden="true"`로 숨김
- 영상/음성 콘텐츠는 **자막/원고/수화 중 최소 1개** 제공
  - 앱 내에 원고 링크라도 제공
- 명도 대비: **최소 3:1, 권장 4.5:1**
  - 텍스트는 가능한 4.5:1 이상 확보
- 지시는 방향/위치 표현 대신 **명칭 기반**으로 안내
  - 예: "오른쪽 버튼" 금지 → "저장 버튼" 사용

## 운용의 용이성(Operable)
- 모든 의미 있는 요소는 **포커스 가능**해야 한다
  - 포커스 이동 순서를 논리적으로 유지
- 제스처 기반 기능은 **동등한 버튼 조작** 제공
  - 드래그/스와이프 전용 기능 금지
- 자동 변경 UI는 **정지/이전/다음** 제공
- 시간 제한이 있으면 **연장/해제** 제공 또는 사전 고지
- 터치 타깃은 **최소 9mm(약 44px)** 확보

## 이해의 용이성(Understandable)
- 폼 입력은 **라벨/힌트/오류 설명** 제공
  - 오류 발생 시 오류 위치로 포커스 이동
- 동일 기능은 동일 UI/동작 유지(일관성)
- 자동재생/깜박임 콘텐츠는 기본 **비활성 또는 제어 UI 제공**
- 외부 앱 이동/팝업/화면 전환은 **사전 예고**

## 견고성(Robust)
- OS 폰트 크기 확대 시 **레이아웃 깨짐 없음**
- 보조기술에서 **role/state/name** 제공
  - 기본 HTML 요소 우선 사용
  - 커스텀 컴포넌트는 ARIA와 키보드 조작 지원 필수

## 앱 공통 규칙
- 버튼/링크는 반드시 명확한 텍스트 또는 `aria-label` 제공
- 토스트/알림은 `aria-live`로 읽히도록 처리
- 모달/팝업은 **포커스 트랩 + 닫힘 시 복귀** 적용
- 입력 오류는 `aria-invalid` + 오류 메시지 `aria-live="polite"`
- 아이콘 버튼은 **텍스트/대체 텍스트** 제공
